⑦⑧ 工具层 + 可观测层

Agent 调用工具的标准接口（MCP）· 驱动 Agent 的 AI 模型 · 看清系统发生了什么（可观测性）

🔌 MCP 协议 — Agent 的 USB 标准 ↗ MCP Servers · ⭐12k+

Agent 需要操作文件、运行命令、控制浏览器——不同工具接口各不相同，整合起来很乱。MCP（Model Context Protocol）就像 USB 接口标准：只要工具实现了 MCP，Agent 就能以同样方式调用任何工具。

没有 MCP（以前）

每个工具有自己的 API 格式，Agent 要写 5 套不同的调用代码，新增工具就要改 Agent 代码。

有了 MCP（现在）

所有工具都说同一种语言，Agent 只需要一套调用方式。新增工具就像插 USB，即插即用。

本项目用到的 5 个 MCP Server

server-filesystem
读写本地文件。Agent 生成的代码通过它落盘，也用它读已有文件来做修复。
官方
server-git
Git 版本控制。每轮生成的代码自动 commit，可以看到修复前后的 diff。
官方
server-sqlite
数据库操作演示。BackendExpert 用它创建表、插入数据，前端演示看得到真实数据。
官方
@playwright/mcp
Web UI 自动化。UIValidator 用它打开浏览器、点击按钮、截图，验证前端界面正确。
官方
desktop-control（自研）
封装 pywinauto，控制 Windows 原生窗口。当生成的是 Tkinter / PyQt 桌面应用时使用。
自研

不同 Agent 用不同模型——为什么？

🎯 Commander DeepSeek-V3 / qwen2.5:14b 需要推理能力强，理解需求、拆任务、做判断
🗄️ BackendExpert DeepSeek-Coder / qwen2.5-coder:7b 纯代码生成，专用代码模型更准
🖥️ FrontendExpert DeepSeek-Coder / qwen2.5-coder:7b 同上，UI 代码生成
🧪 TestExpert DeepSeek-Coder / qwen2.5-coder:7b 测试代码生成，逻辑相对简单
🔍 Validator DeepSeek-V3 / qwen2.5:14b 需要理解验收标准并对比，推理要求高
🧠 mem0 内部 qwen2.5:7b（Ollama） 纯本地运行，提炼记忆不需要最强模型

📡 可观测层 — 看清 Agent 在做什么

Agent 系统出问题很难 debug：哪个节点卡住了？哪次 LLM 调用耗时最长？Token 烧了多少？可观测层回答这三个问题。

OpenTelemetry ↗ SDK
分布式追踪埋点标准
在代码里埋点，记录「谁调用了谁、花了多少时间、成功还是失败」。不负责展示，只负责采集。
类比：在每通电话里 装一个录音机
Jaeger ↗ ⭐20k+
链路追踪可视化 UI
接收 OpenTelemetry 发来的追踪数据，画成时间轴图。一眼看出哪个节点慢、哪里报错。
类比：把录音整理成 通话记录时间轴
LangSmith ↗ SaaS
LLM 专项追踪
专门记录每次 LLM 调用：发了什么 Prompt、AI 回了什么、用了多少 Token、花了多少时间。
类比：专门记录「对 AI 说了什么、AI 答了啥」

三者分工

# OpenTelemetry 埋点（写在代码里，自动采集） with tracer.start_as_current_span("commander.decompose") as span: result = commander.run(requirement) span.set_attribute("task_count", len(result.tasks)) # ↑ 这段数据会自动发给 Jaeger，在 UI 里显示成时间轴 # LangSmith 自动追踪（只需配置环境变量） LANGCHAIN_TRACING_V2="true" LANGCHAIN_API_KEY="ls_..." # ↑ 之后所有 LangChain/LangGraph 的 LLM 调用自动上报到 LangSmith
