⑥ 验证者 Agent · 闭环修复

不满意就打回重做——让整个系统从「一次性生成」变成「持续改进」。

🏭 类比：工厂质检员

专家池就像流水线工人，写完代码放到传送带上。验证者像质检员站在出口：合格的放行，不合格的贴上问题标签打回去重做，最多返工 5 次，还不行就标「尽力而为」出厂。

没有验证者，AI 写出的代码哪怕有 bug 也直接交付。有了验证者，系统才有「自我纠错」能力。

🔍 验证者做四项检查

① 代码质量
ruff 静态检查
自动扫描代码：语法错误、未使用变量、命名不规范……比 pylint 快 10 倍，秒出结果。
E501 line too long (92 > 88) F401 'os' imported but unused → 2 errors found
② 功能正确性
pytest 测试结果
读取 TestExpert 已经跑完的测试结果，看有没有 failed。不是重新跑，是读现成结论。
test_create_todo PASSED ✅ test_delete_todo FAILED ❌ → 1 failed, 1 passed
③ 验收标准对比
LLM 逐条核对
Commander 最初定了验收标准（比如「支持添加、完成、删除」），LLM 逐条检查代码里有没有实现。
✅ 添加任务 — 已实现 ✅ 完成任务 — 已实现 ❌ 删除任务 — 函数不存在
④ 界面验证
截图结果读取
读取 UIValidator 的截图检查结论，确认「添加按钮」「任务列表」等 UI 元素存在。
✅ add-button 存在 ✅ todo-list 存在 ❌ delete-button 未找到

🚦 核心机制：条件边（Conditional Edge）

LangGraph 里，验证者的输出决定下一步去哪——这就叫条件边。普通边是固定的「A → B」，条件边是「A → B 或 C，取决于结果」。

Validator 检查
✅ 全部通过
·
❌ 有问题 & 轮次 < 5
·
⛔ 轮次 = 5
→ 完成，推送到前端
→ 打回 Expert 修复
→ 强制结束（尽力而为）
# LangGraph 条件边的实际代码 def should_continue(state: ProjectState) -> str: result = state["validation_result"] # 超过 5 轮强制结束，不管通没通过 if state["iteration_count"] >= 5: return "done" # 四项全通过 → 完成 if result.all_passed: return "done" # 有问题 → 打回修复 return "fix" # 在 StateGraph 里注册条件边 graph.add_conditional_edges( "validator", # 从这个节点出发 should_continue, # 用这个函数判断 { "done": END, # "done" → 结束 "fix": "fix_expert", # "fix" → 去修复节点 } )

🛡️ 循环保护：最多 5 轮

AI 修了一遍，可能还是有问题，再修，再检查……如果没有上限，这个循环永远不会停。iteration_count 就是计数器，超过 5 强制结束。

⚙️ 一次完整的「检查 → 打回 → 修复」流程

1
专家池完成代码，写入 ProjectState。Validator 开始四项检查。
2
发现问题：ruff 报告 F401 未使用变量；pytest 有 1 个 failed；LLM 发现「删除任务」函数缺失。
3
Validator 把问题汇总，写入 state["fix_feedback"]：
「删除函数缺失；test_delete_todo 失败；未使用变量 os」
4
条件边判断 → 返回 "fix" → LangGraph 路由到 fix_expert 节点。
iteration_count 从 1 变成 2。
5
BackendExpert 读取 fix_feedback，带着问题描述重新生成，补上删除函数，删掉未用变量。
6
Validator 第二轮检查：四项全通过 → 条件边返回 "done" → 推送结果到前端 ✅
