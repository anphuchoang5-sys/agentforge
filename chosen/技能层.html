⑤ Skills 技能层 · 三层渐进式加载

Agent 的「按需加载工具书」——用到哪页翻哪页，不浪费上下文。

🤔 为什么不直接把所有指令塞给 AI？

Agent 做每件事都需要「指导手册」——怎么写规格、怎么拆任务、怎么写代码……如果一次性全部加载，token 爆炸：

❌ 一次性全加载

6 个技能 × 5000 token = 30,000 token
光看手册就把脑容量占满，还没开始干活

✅ 三层渐进式加载

启动时只加载 6 × 100 = 600 token
用到哪个技能才加载完整内容，用完即释放

📱 三层加载（类比手机 App）

🏠
L1 发现层
桌面图标
只看到 App 名字，几乎不占空间
📱
L2 激活层
打开 App
加载完整界面和功能说明
⚙️
L3 穿透层
使用某功能
加载该功能的具体数据和脚本
L1
发现层 — 系统启动时加载
每个技能只有名字 + 一句描述，让 AI 知道「有哪些技能可以用」。6 个技能全部常驻，几乎不占上下文。
~100
token/技能
L2
激活层 — 任务匹配时加载
判断当前任务需要哪个技能，加载该技能的完整指令、参数说明、注意事项。用完即释放，不同时保留多个。
<5000
token/技能
L3
穿透层 — 实际执行时加载
执行时才加载具体脚本、代码模板、Schema 定义、业务规范文件。执行完毕立即释放。
按需
动态加载

🛠️ 六个标准技能 — 一条完整的开发闭环

按顺序走完就是一个从「需求」到「交付」的完整软件开发流程：

/spec
需求规格
先定「做什么」，再写代码。输出接口规范、验收标准。
「没有规格的代码，是在猜测需求」
/plan
任务规划
把规格拆成原子任务，每个任务只做一件事，输出 DAG 依赖图。
「拆得越细，出错越少」
/build
代码构建
一次只实现一个功能切片，不贪多，写完要能运行。
「每次只咬一口，不要贪大」
/test
测试生成
写测试证明功能可用，不是「可能可用」。测试是功能存在的证明。
「没有测试的功能不算完成」
/review
代码审查
审查命名、结构、边界情况，提升代码健康度，不是找 bug。
「好代码是改出来的」
/ship
交付打包
生成文档、打包部署、确认所有验收标准通过，完成交付。
「跑通才算完成」

⚙️ 一次完整任务的技能调用过程

# 系统启动
L1 加载全部技能图标（600 token 常驻）
✓ /spec /plan /build /test /review /ship

# Commander 收到「开发待办 App」，判断需要哪些技能
→ 需要 /spec + /plan + /build + /test

L2 激活 /spec（4000 token）→ 生成接口规范 → 释放
L2 激活 /plan（3500 token）→ 生成任务 DAG → 释放

# BackendExpert 开始
L2 激活 /build（4500 token）→ 开始写数据库代码
L3 加载 scripts/build_template.py → 执行 → 释放
→ 写完，/build 释放

# TestExpert 开始
L2 激活 /test（3000 token）→ 写 pytest 测试
L3 加载 scripts/run_pytest.py → 执行 → 释放
→ 5 passed ✅，/test 释放

# 上下文始终精简，峰值不超过 L1(600) + L2(5000) = 5600 token 用于 Skills

📁 Skill 文件结构

每个技能是一个文件夹，SKILL.md 控制三层加载内容：

skills/
├── spec/
│ ├── SKILL.md ← L1 描述 + L2 完整指令都在这
│ ├── scripts/ ← L3：可执行脚本
│ ├── references/ ← L3：按需加载的业务规范
│ └── assets/ ← L3：Schema 模板文件
├── plan/
├── build/
├── test/
├── review/
└── ship/
# SKILL.md 内部结构（控制三层加载） --- name: build l1_description: "一次实现一个功能切片" # L1 常驻 l2_instructions: | # L2 按需加载 只实现当前任务，不要超出范围 每次提交只做一件事 写完要能运行，不允许留 TODO l3_scripts: ["scripts/run_build.py"] # L3 执行时加载 ---
